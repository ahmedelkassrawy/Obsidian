# MCP Architecture

MCP (Model Context Protocol) is a standard, like HTTP or USB-C, for connecting AI applications to external tools and data sources. Consistent terminology is essential for clear system design and interoperability.

## Core terminology

### Components

MCP has a client-server architecture:

- **Host**: The user-facing AI application. Examples: Claude Desktop, AI IDEs, SDK-based applications (LangChain/smolagents), or custom apps.
- **Client**: A component inside the host that manages communication with one MCP server (1:1 per connection).
- **Server**: An external program/service exposing MCP capabilities (Tools, Resources, Prompts, and Sampling support).

> **Important:** Host and Client are not the same. The Host is the full application; the Client is the protocol component inside it.

### Capability overview

| Capability | Description | Example |
| --- | --- | --- |
| **Tools** | Executable functions the model can invoke for actions or computed data. | A weather function for a specific location. |
| **Resources** | Read-only context/data sources with minimal computation. | A scientific papers corpus. |
| **Prompts** | Predefined templates/workflows guiding interactions. | A summarization template. |
| **Sampling** | Server-initiated request for Host/Client to run LLM inference. | A writing workflow that asks for iterative refinement. |

## Communication flow

1. **User interaction**: User sends intent/query to the **Host**.
2. **Host processing**: Host interprets intent (often with an LLM) and decides needed capabilities.
3. **Client connection**: Host delegates to its **Client** to connect to relevant server(s).
4. **Capability discovery**: Client discovers available server capabilities.
5. **Capability invocation**: Host directs Client to invoke required capabilities.
6. **Server execution**: Server executes and returns output to Client.
7. **Result integration**: Client returns results to Host for display and/or further LLM reasoning.

### Request

A request is sent from Client to Server to initiate an operation. It includes:
- A unique `id`
- A `method` (for example, `tools/call`)
- Method `params`

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "weather",
    "arguments": {
      "location": "San Francisco"
    }
  }
}
```

### Response

A response is sent from Server to Client for a prior request. It includes:
- The same `id`
- Either a `result` or an `error`

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "temperature": 62,
    "conditions": "Partly cloudy"
  }
}
```

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "error": {
    "code": -32602,
    "message": "Invalid location parameter"
  }
}
```

### Notification

A notification is a one-way message (no response required), typically for progress/events.

```json
{
  "jsonrpc": "2.0",
  "method": "progress",
  "params": {
    "message": "Processing data...",
    "percent": 50
  }
}
```

## Capability details

### Tools

```python
def get_weather(location: str) -> dict:
    """Get the current weather for a specified location."""
    # Connect to weather API and fetch data
    return {
        "temperature": 72,
        "conditions": "Sunny",
        "humidity": 45
    }
```

### Resources

Resources provide read-only data access.
- **Control**: Application/Host controlled
- **Nature**: Retrieval-focused (similar to REST `GET`)
- **Safety**: Lower risk than Tools due to no direct side effects

```python
def read_file(file_path: str) -> str:
    """Read the contents of a file at the specified path."""
    with open(file_path, "r") as f:
        return f.read()
```

### Prompts

Prompts are predefined templates/workflows.

```python
def code_review(code: str, language: str) -> list:
    """Generate a code review for the provided code snippet."""
    return [
        {
            "role": "system",
            "content": f"You are a code reviewer examining {language} code. Provide a detailed review highlighting best practices, potential issues, and suggestions for improvement."
        },
        {
            "role": "user",
            "content": f"Please review this {language} code:\n\n```{language}\n{code}\n```"
        }
    ]
```

### Sampling

Sampling allows Servers to request the Client/Host to perform LLM interactions.
- **Control**: Server-initiated, Host/Client-mediated
- **Purpose**: Enables multi-step and agentic behaviors

## 1. How Sampling Works

Usually, the flow is: **User → Host (Claude/Gemini) → Server (Tool).** With sampling, the flow can become: **Host → Server → (asks Host for help) → Host.**

When a server uses the `sampling/createMessage` method, it asks the host to:
1. Take a specific prompt or set of messages.
    
2. Run them through the LLM.
    
3. Return the text or tool call result back to the server.

```python
def request_sampling(messages, system_prompt=None, include_context="none"):
    """Request LLM sampling from the client."""
    # In a real implementation, this would send a request to the client
    return {
        "role": "assistant",
        "content": "Analysis of the provided data..."
    }
```

#### Sampling flow

1. Server sends `sampling/createMessage` to the Client.
2. Client reviews and may modify the request.
3. Client samples from an LLM.
4. Client reviews the completion.
5. Client returns the result to the Server.

## Capability comparison

| Capability | Controlled By | Direction | Side Effects | Approval Needed | Typical Use Cases |
| --- | --- | --- | --- | --- | --- |
| Tools | Model (LLM) | Client -> Server | Yes (potentially) | Yes | Actions, API calls, data manipulation |
| Resources | Application | Client -> Server | No (read-only) | Typically no | Data retrieval, context gathering |
| Prompts | User | Server -> Client | No | No (user-selected) | Guided workflows, specialized templates |
| Sampling | Server | Server -> Client -> Server | Indirectly | Yes | Multi-step tasks, agentic behavior |

These capabilities complement each other:
1. User picks a **Prompt** to start a workflow.
2. The workflow pulls context from **Resources**.
3. The model invokes **Tools** for actions.
4. The server can request **Sampling** for additional LLM reasoning.

## Discovery endpoints

```text
- `tools/list`: Discover available Tools
- `resources/list`: Discover available Resources
- `prompts/list`: Discover available Prompts
```

## Minimal FastMCP example

```python
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("weather")

@mcp.tool()
def get_weather(location: str) -> str:
    """Get the current weather for a given location."""
    return f"The current weather in {location} is sunny with a temperature of 25°C."

@mcp.resource("weather://{location}")
def weather_resource(location: str) -> str:
    """Provide weather data as a resource."""
    return f"Weather data for {location}: Sunny, 72°F"

@mcp.prompt()
def weather_report(location: str) -> str:
    """Create a weather report prompt."""
    return f"You are a weather reporter. Weather report for {location}?"

if __name__ == "__main__":
    mcp.run()
```
