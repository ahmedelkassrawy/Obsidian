# AI Contract Review — Advanced Temporal Patterns

## Overview

This project builds on the basic PDF pipeline and introduces **four new Temporal concepts**:

1. **Child Workflows** — fan out parallel work
2. **Queries** — read workflow state from outside
3. **Signals** — send data into a running workflow
4. **Updates** — send data **and wait for validation + response**

The system ingests multiple PDF contracts, extracts and summarizes each one via a child workflow, synthesizes a cross-contract risk report, then **pauses for human review** before finalizing.

## 2. Child Workflows (Fan-Out)

A **child workflow** is a workflow started by another workflow. The parent can start many children in parallel and collect their results.

### Child Workflow — `PDFSummaryWorkflow`

This handles **one PDF**: extract text, summarize via LLM.

```python
# child_workflow.py
@workflow.defn
class PDFSummaryWorkflow:
    @workflow.run
    async def run(self, params: PDFSummaryInput) -> PDFSummaryOutput:
        # Step 1: extract PDF text
        extract_text = await workflow.execute_activity(
            extract_pdf,
            ExtractPDFInput(s3_path=params.s3_path),
            start_to_close_timeout=timedelta(minutes=10),
            heartbeat_timeout=timedelta(seconds=60),
        )

        # Step 2: call LLM to summarize
        llm_result = await workflow.execute_activity(
            call_llm,
            LLMTextInput(text=extract_text.markdown_text),
            start_to_close_timeout=timedelta(minutes=10),
        )

        parsed = json_repair.loads(llm_result.content)
        return PDFSummaryOutput(
            s3_path=extract_text.s3_path,
            summary=parsed.get("summary", ""),
            key_risks=parsed.get("key_risks", ""),
        )
```

### Parent launches children in parallel

The parent uses `asyncio.gather()` to start **N child workflows concurrently**:

```python
# parent_workflow.py (inside ContractReviewWorkflow.run)
raw_results = await asyncio.gather(
    *[
        workflow.execute_child_workflow(
            PDFSummaryWorkflow.run,
            PDFSummaryInput(s3_path=current_s3_path),
            id=f"{workflow_id}-pdf-{idx + 1}",
            task_queue=workflow_task_queue,
            parent_close_policy=ParentClosePolicy.ABANDON,
        )
        for idx, current_s3_path in enumerate(params.s3_paths)
    ],
    return_exceptions=True,
)
```

| Concept | What it means |
|---|---|
| `workflow.execute_child_workflow()` | Starts a child workflow and awaits its result |
| `id=f"{id}-pdf-{idx}"` | Each child gets a unique, predictable ID — visible in the Temporal UI |
| `ParentClosePolicy.ABANDON` | If the parent is cancelled/terminated, children keep running |
| `return_exceptions=True` | Exceptions from individual children don't crash the whole `gather` — they're returned as `Exception` objects, letting the parent handle partial failures gracefully |

### Fan-out flow

```
Parent                              Children
  │                                      │
  ├── execute_child_workflow(PDF 1) ────▶│ (starts PDFSummaryWorkflow)
  │                                      ├── extract_pdf()
  │                                      ├── call_llm()
  │                                      └── returns result
  │◀──────────── result ─────────────────┤
  │                                      │
  ├── execute_child_workflow(PDF 2) ────▶│
  │                                      ├── extract_pdf()
  │                                      ├── call_llm()
  │                                      └── returns result
  │◀──────────── result ─────────────────┤
  │                                      │
  ├── execute_child_workflow(PDF 3) ────▶│
  │                                      ├── ...
  │◀──────────── result ─────────────────┤
  │                                      │
  │  (all done — now synthesize)         │
```

**Why not just use activities?** You could, but child workflows give you:
- Independent retry and timeout policies per PDF
- Each child has its own event history (debuggable in the Temporal UI)
- A child can itself contain multiple activities (extract → summarize)

## 3. LLM Synthesis (in the parent)

After all children return, the parent feeds their summaries into a **synthesis prompt** that identifies cross-contract risks:

```python
# parent_workflow.py — after fan-out
combined_summary = "\n\n".join([
    f"**Contract {i+1}** (`{summary['s3_path']}`):\n"
    f"Summary: {summary['summary']}\n"
    f"Risks: {summary['key_risks']}"
    for i, summary in enumerate(self._summaries)
])

llm_result = await workflow.execute_activity(
    call_llm,
    LLMInput(prompt=SYNTHESIS_PROMPT.format(
        summaries=combined_summary, n=len(self._summaries)
    )),
    start_to_close_timeout=timedelta(minutes=3),
)

parsed = json_repair.loads(llm_result.content)
self._report = parsed
```

The synthesis prompt (`prompts.py`) asks the LLM for:
- `overall_risk_level`
- `top_cross_contract_risks`
- `recommended_actions`

## 4. Human-in-the-Loop (HITL)

This is the core of the document. Temporal makes HITL natural because workflows can **pause indefinitely** waiting for external input.

### The pattern

```python
# parent_workflow.py — HITL loop
for revision_no in range(params.max_revision + 1):
    self._status = "awaiting-review"
    self._review_decision = None

    # ── PAUSE here until a human sends a decision ──
    try:
        await workflow.wait_condition(
            lambda: self._review_decision is not None,
            timeout=timedelta(days=3),  # safety: auto-complete after 3 days
        )
    except asyncio.TimeoutError:
        break

    if self._review_decision == "approve":
        break  # exit loop → complete

    # ── Revise: send feedback to LLM, regenerate ──
    revised_report = await workflow.execute_activity(
        call_llm,
        LLMInput(prompt=REVISION_PROMPT.format(
            report=json.dumps(self._report, indent=2),
            feedback=self._review_feedback,
        )),
        ...
    )
    self._report = json_repair.loads(revised_report.content)
    # loop back to awaiting-review
```

### Three mechanisms that make HITL work

Temporal provides three ways to communicate with a running workflow from **outside** (e.g., from an API client):

| Mechanism | Direction | Returns data? | Validated? | Use case |
|---|---|---|---|---|
| **Query** | Read-only | ✅ Yes | ❌ No | Get current status or report |
| **Signal** | One-way write | ❌ No | ❌ No | Assign a reviewer name |
| **Update** | Write + response | ✅ Yes | ✅ Yes | Submit approve/revise decision |

## 5. Queries — Reading State

A **Query** exposes workflow state without modifying it. Queries are handled **immediately** from the workflow's in-memory state — they don't create new event history.

### Defining queries

```python
# parent_workflow.py
@workflow.query
def get_status(self) -> dict:
    return {
        "status": self._status,
        "pdf_processed": len(self._summaries),
        "report_preview": str(self._report)[:500],
        "approved_by": self._approved_by,
    }

@workflow.query
def get_report(self) -> dict:
    return {
        "status": self._status,
        "report": self._report,
        "approved_by": self._approved_by,
        "sources": [s["s3_path"] for s in self._summaries],
    }
```

### Calling queries from the client

```python
# client-app/main.py
workflow_state = await handle.query("get_status", result_type=dict)
workflow_report = await handle.query("get_report", result_type=dict)
```

### API endpoints

```python
# FastAPI
@app.get("/contract-review/{workflow_id}/status")
async def get_review_status(workflow_id: str):
    handle = client.get_workflow_handle(workflow_id)
    desc = await handle.describe()
    if desc.status == WES.RUNNING:
        state = await handle.query("get_status", result_type=dict)
    return {"workflow_id": workflow_id, "workflow_state": state}

@app.get("/contract-review/{workflow_id}/report")
async def get_review_report(workflow_id: str):
    # same pattern but calls the "get_report" query
```

**Why queries instead of just reading from a database?** Because the workflow state is **live** — you're reading the current in-memory state of a running workflow, not a stale snapshot.

## 6. Signals — Writing Data (Fire-and-Forget)

A **Signal** is a one-way message into a workflow. The sender doesn't wait for a response.

### Defining a signal

```python
# parent_workflow.py
@workflow.signal
async def assign_reviewer(self, name: str) -> None:
    self._approved_by = name
```

### Sending a signal

```python
# client-app/main.py
@app.post("/contract-review/{workflow_id}/assign")
async def assign_reviewer(workflow_id: str, request: AssignRequest):
    handle = client.get_workflow_handle(workflow_id)
    await handle.signal("assign_reviewer", request.name)
    return {"message": f"Reviewer '{request.name}' assigned."}
```

Signal is ideal for **non-critical, fire-and-forget updates** like recording who is reviewing.

> **Signal vs Update**: A signal cannot return data or validate input. Use signals for simple notifications; use updates when you need validation or a response.

## 7. Updates — Writing Data with Validation + Response

An **Update** is like a signal that returns a result. It also supports **validators** that run before the update is applied.

### Defining an update

```python
# parent_workflow.py
@workflow.update
async def submit_decision(self, decision: str, feedback: str = "") -> str:
    self._review_decision = decision
    self._review_feedback = feedback
    return f"Decision '{decision}' recorded."

@submit_decision.validator
def validate_decision(self, decision: str, feedback: str = "") -> None:
    if decision not in ("approve", "revise"):
        raise ValueError(f"Must be 'approve' or 'revise', got: '{decision}'")
    if decision == "revise" and not feedback.strip():
        raise ValueError("Feedback is required when requesting a revision.")
```

### Calling an update

```python
# client-app/main.py
@app.post("/contract-review/{workflow_id}/revise")
async def submit_revise(workflow_id: str, request: ReviseRequest):
    handle = client.get_workflow_handle(workflow_id)
    result = await handle.execute_update(
        "submit_decision",
        args=["revise", request.feedback],
    )
    return {"ok": True, "message": result}

@app.get("/contract-review/{workflow_id}/approve")
async def submit_approve(workflow_id: str):
    handle = client.get_workflow_handle(workflow_id)
    result = await handle.execute_update(
        "submit_decision",
        args=["approve", ""],
    )
    return {"ok": True, "message": result}
```

### How `wait_condition` connects to all this

The workflow pauses at `await workflow.wait_condition(lambda: self._review_decision is not None)`. This is a **blocking await** inside the workflow that suspends execution until the condition becomes true.

When a client sends an **Update** (`submit_decision("approve")`):
1. The Temporal server receives the update
2. The validator runs (checks `decision in ("approve", "revise")`)
3. If valid, the update handler sets `self._review_decision = "approve"`
4. The `wait_condition` re-evaluates — now `self._review_decision is not None` is **True**
5. The workflow **unblocks** and continues

This is the core HITL pattern — a workflow that pauses indefinitely and resumes when external input arrives.

## 8. Worker Registration

The worker registers **all** workflows and activities for both the parent and child:

```python
# worker.py
worker = Worker(
    temporal_client,
    task_queue=TEMPORAL_TASK_QUEUE,
    workflows=[AIContractReviewWorkflow, PDFSummaryWorkflow],
    activities=[extract_pdf, call_llm],
)
```

Both workflows share the same task queue. When the parent calls `execute_child_workflow(..., task_queue="contract-review-queue")`, the child tasks go to the same queue and the **same worker** picks them up.

## 10. Most Common HITL Bug

The HITL code **must** be inside the `run()` method **before** the final `return`. A `return` placed too early makes the entire HITL loop dead code:

```python
# ❌ WRONG — HITL never runs
parsed = json_repair.loads(llm_result.content)
self._report = str(parsed)

return ContractReviewOutput(...)    # ← Workflow ends here

for revision_no in ...:             # ← Unreachable!
    await workflow.wait_condition(...)
```

```python
# ✅ CORRECT — HITL runs after synthesis
parsed = json_repair.loads(llm_result.content)
self._report = parsed

for revision_no in range(params.max_revision + 1):
    await workflow.wait_condition(...)   # ← Now this actually runs
    ...

return ContractReviewOutput(...)        # ← Return at the very end
```

## Concepts Summary

| Concept | What it does | Analogy |
|---|---|---|
| **Child Workflow** | A workflow started by another workflow | Manager delegates to team leads |
| **Fan-out** | Start many child workflows in parallel | Parallel `asyncio.gather()` for workflows |
| **ParentClosePolicy** | Controls what happens to children if parent dies | `ABANDON` = children live on |
| **Query** | Read current workflow state | GET endpoint but live |
| **Signal** | Send data into a workflow (fire-and-forget) | POST with no return |
| **Update** | Send data + get response + validation | POST with validation + return |
| **Validator** | Runs before an update is applied | Pydantic validation on a PUT |
| **wait_condition** | Pause workflow until a predicate is true | `await` with a callback |
| **HITL** | Human-in-the-Loop — pause for external decision | Approval workflow |
