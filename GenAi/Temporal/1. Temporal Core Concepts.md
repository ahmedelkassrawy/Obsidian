# Temporal Core Concepts

## Overview

Temporal is a **durable execution platform** — it lets you write asynchronous code as a **workflow** composed of **activities**, and guarantees it runs to completion even if the worker crashes halfway through.

This document explains every concept through a real project: a PDF pipeline that downloads a PDF from S3, extracts markdown text, and uploads the result back to S3.

---

## Architecture

```
┌─────────────┐     ┌──────────────────┐     ┌──────────────────────┐
│   Client    │────▶│  Temporal Server │◀───▶│       Worker         │
│  (FastAPI)  │     │  (orchestration) │     │  (poll + execute)    │
└─────────────┘     └──────────────────┘     └──────────────────────┘
                                                    │
                                           ┌─────────┼──────────┐
                                           ▼                   ▼
                                   ┌──────────────┐   ┌──────────────┐
                                   │  Activity 1  │   │  Activity 2  │
                                   │ download_pdf │   │extract_to_...│
                                   └──────────────┘   └──────────────┘
```

## 1. Temporal Server

The server is the backbone. It stores workflow state, manages task queues, handles retries, and ensures durability.

**Setup** (Docker Compose):
```yaml
# docker-compose.yml (official Temporal sample)
services:
  temporal:
    image: temporalio/auto-setup:latest
    ports:
      - "7233:7233"
  temporal-ui:
    image: temporalio/ui:latest
    ports:
      - "8080:8080"
```

## 2. Client — Starting a Workflow

The **client** connects to the Temporal server and kicks off a workflow. It can **execute** (wait for result) or **start** (fire-and-forget with a handle).

```python
from temporalio.client import Client

client = await Client.connect("localhost:7233", namespace="default")
```

### Execute (blocking — waits for result)

```python
# client-app/main.py
results = await client.execute_workflow(
    "PDFPipelineWorkflow",
    args=[{"s3_path": "s3://bucket/file.pdf"}],
    id="pdf-pipeline-<uuid>",
    task_queue="pdf-pipeline-queue",
    result_type=dict,
)
```

### Start (non-blocking — returns handle)

```python
# client-app/main.py
handle = await client.start_workflow(
    "PDFPipelineWorkflow",
    args=[{"s3_path": "s3://bucket/file.pdf"}],
    id="pdf-pipeline-<uuid>",
    task_queue="pdf-pipeline-queue",
)

# Later — poll for status or result
desc = await handle.describe()
result = await handle.result()
```

| Method | Returns | Use case |
|---|---|---|
| `execute_workflow()` | The workflow result | Block until done |
| `start_workflow()` | `WorkflowHandle` | Fire-and-forget, check later |

## 3. Task Queue

The **task queue** is the channel between the client and the worker. The client puts a task on the queue; the worker polls that queue.

Both sides must use the **same queue name**:

```python
# In the client (main.py)
task_queue = "pdf-pipeline-queue"

# In the worker (worker.py)
Worker(
    client,
    task_queue="pdf-pipeline-queue",   # ← same name
    workflows=[PDFPipelineWorkflow],
    activities=[download_pdf, ...],
)
```

---

## 4. Worker

A **worker** is a long-running process that polls the Temporal server for tasks and executes them. It must register both workflows and activities.

```python
# worker.py
async def main():
    client = await Client.connect("localhost:7233", namespace="default")

    worker = Worker(
        client,
        task_queue="pdf-pipeline-queue",
        workflows=[PDFPipelineWorkflow],
        activities=[download_pdf, extract_to_markdown, upload_markdown],
    )

    await worker.run()

if __name__ == "__main__":
    asyncio.run(main())
```

One worker can handle **many workflow instances** — Temporal multiplexes them automatically.

## 5. Workflow

A **workflow** is the orchestration logic. It is:
- **Durable** — its state is persisted after every `await`
- **Fault-tolerant** — if the worker dies, another picks up from the last `await`
- **Deterministic** — no random, no direct I/O (that's what activities are for)

```python
# workflow_process_pdf.py
@workflow.defn
class PDFPipelineWorkflow:

    @workflow.run
    async def run(self, params: PDFPipelineInput) -> PDFPipelineOutput:
        # Step 1
        download_result = await workflow.execute_activity(
            download_pdf,
            DownloadInput(s3_path=params.s3_path),
            start_to_close_timeout=timedelta(minutes=3),
            retry_policy=DEFAULT_RETRY_POLICY,
        )

        # Step 2
        extract_result = await workflow.execute_activity(
            extract_to_markdown,
            ExtractInput(local_pdf_path=download_result.local_path),
            start_to_close_timeout=timedelta(minutes=3),
        )

        # Step 3
        upload_result = await workflow.execute_activity(
            upload_markdown,
            UploadMarkdownInput(...),
            start_to_close_timeout=timedelta(minutes=3),
        )

        return PDFPipelineOutput(output_s3_path=upload_result.s3_path)
```

### Key rules for workflows
- ❌ No `asyncio.sleep()` — use `workflow.sleep()` instead
- ❌ No `random` — use `workflow.random()`
- ❌ No direct HTTP/S3 calls — delegate to activities
- ✅ Use `workflow.execute_activity()` for any I/O
- ✅ All `await` points become **event history checkpoints**

---

## 6. Activity

An **activity** is a single unit of work — one function that does one thing (download, extract, upload). Activities **can** do I/O, sleep, and use randomness freely.

```python
# activities.py
@activity.defn
async def download_pdf(params: DownloadInput) -> DownloadOutput:
    bucket, key = parse_s3_path(params.s3_path)
    local_path = str(Path(TEMP_DIR) / Path(key).name)

    activity.logger.info(f"Downloading {params.s3_path} to {local_path}")

    s3_client = get_s3_client()
    s3_client.download_file(bucket, key, local_path)

    return DownloadOutput(local_path=local_path)
```

```python
@activity.defn
async def extract_to_markdown(params: ExtractInput) -> ExtractOutput:
    markdown_text = pymupdf4llm.to_markdown(params.local_pdf_path)
    return ExtractOutput(markdown_text=markdown_text)
```

```python
@activity.defn
async def upload_markdown(params: UploadMarkdownInput) -> UploadMarkdownOutput:
    bucket, key = parse_s3_path(params.original_s3_path)
    markdown_key = key.replace(".pdf", ".md")
    s3_client.put_object(Bucket=bucket, Key=markdown_key, Body=...)
    return UploadMarkdownOutput(s3_path=f"s3://{bucket}/{markdown_key}")
```

### I/O boundary
- **Workflows** → orchestration only (pure coordination)
- **Activities** → all side effects (S3, filesystem, APIs)

This separation is what makes Temporal powerful: the workflow is replayable from event history because all non-determinism stays inside activities.

## 7. Input/Output with Dataclasses

Temporal serializes inputs and outputs to/from JSON automatically. Use **dataclasses** for type safety and self-documenting code.

```python
# helpers.py
@dataclass
class DownloadInput:
    s3_path: str

@dataclass
class DownloadOutput:
    local_path: str

@dataclass
class ExtractInput:
    local_pdf_path: str

@dataclass
class ExtractOutput:
    markdown_text: str
```

The workflow passes these between activities:
```python
download_result = await workflow.execute_activity(
    download_pdf,
    DownloadInput(s3_path=params.s3_path),   # ← typed input
    ...
)
# download_result is a DownloadOutput with .local_path
```

## 8. Retry Policy

**Automatic retries** on failure. You configure the backoff:

```python
# workflow_process_pdf.py
from temporalio.common import RetryPolicy

DEFAULT_RETRY_POLICY = RetryPolicy(
    initial_interval=timedelta(seconds=5),    # wait 5s before first retry
    backoff_coefficient=2.0,                   # double each time (5 → 10 → 20 → ...)
    maximum_interval=timedelta(seconds=60),    # cap at 60s
    maximum_attempts=5,                        # give up after 5 tries
)
```

Apply to any activity:
```python
await workflow.execute_activity(
    download_pdf,
    DownloadInput(...),
    start_to_close_timeout=timedelta(minutes=3),
    retry_policy=DEFAULT_RETRY_POLICY,  # ← retries on failure
)
```

If an activity fails (e.g., S3 404), Temporal retries automatically with exponential backoff. After 5 attempts, the workflow itself fails — and you can inspect why in the Temporal UI.

---

## 9. Timeouts

Two critical timeouts:

| Timeout | Meaning | Example |
|---|---|---|
| `start_to_close_timeout` | Max time from when activity starts until it finishes | 3 minutes for a PDF download |
| `schedule_to_close_timeout` | Max time from when activity is scheduled until it finishes (includes queue wait) | Longer — accounts for backlog |
| `schedule_to_start_timeout` | Max time the activity can sit on the queue before a worker picks it up | |

```python
start_to_close_timeout=timedelta(minutes=3)
```

If an activity exceeds its timeout, Temporal marks it failed and retries (if configured).

## 10. Workflow Sandbox (Safe Imports)

Temporal runs workflow code in a **sandbox** to enforce determinism. Activities that do I/O cannot be imported directly inside a workflow file — you must use `unsafe.imports_passed_through()`:

```python
# workflow_process_pdf.py
with workflow.unsafe.imports_passed_through():
    from activities import (
        download_pdf,
        extract_to_markdown,
        upload_markdown,
    )
    from helpers import DownloadInput, ExtractInput, UploadMarkdownInput
```

This tells Temporal: "I know these imports have side effects — allow them." The workflow itself still runs in a deterministic replay context.

## Putting It All Together

### The full flow

```
Client                          Temporal Server                    Worker
  │                                    │                              │
  ├── execute_workflow() ─────────────▶│                              │
  │   (s3://bucket/file.pdf)           │                              │
  │                                    ├── schedule task ────────────▶│
  │                                    │   (task queue)               │
  │                                    │                              ├── download_pdf()
  │                                    │                              │   (S3 → local file)
  │                                    │◀────────── result ──────────┤
  │                                    │                              ├── extract_to_markdown()
  │                                    │                              │   (PDF → markdown text)
  │                                    │◀────────── result ──────────┤
  │                                    │                              ├── upload_markdown()
  │                                    │                              │   (markdown → S3 .md)
  │                                    │◀────────── result ──────────┤
  │◀────────── workflow result ────────┤                              │
```

### All 4 components in the codebase

| File | Role |
|---|---|
| `apps/pdf-extraction-plain/process_pdf.py` | Plain Python version (no Temporal) — same logic but no durability |
| `apps/pdf-extraction-temporal/worker.py` | Worker — polls queue, runs registered workflows/activities |
| `apps/pdf-extraction-temporal/workflow_process_pdf.py` | Workflow — orchestration (3 steps, retries, timeouts) |
| `apps/pdf-extraction-temporal/activities.py` | Activities — actual S3 download/extract/upload |
| `apps/pdf-extraction-temporal/helpers.py` | Shared dataclasses + S3 helpers |
| `apps/client-app/main.py` | FastAPI client — starts workflows via HTTP endpoint |

## Why Temporal Over Plain Code?

| Concern | Plain script | Temporal |
|---|---|---|
| Worker crashes mid-pipeline | Lose progress, start over | Resumes from last `await` |
| Retry on transient failure | Manual try/except loops | Declarative `RetryPolicy` |
| Monitoring | Logs only | Web UI at localhost:8080 |
| Long-running pipelines | Blocking process | Async, workers can restart |
| Step-by-step visibility | Print statements | Event history per workflow |
