# Data Engineering

## Introduction to Data Storage and Structure

Understanding how data is stored and structured is critical for effective data management in machine learning (ML) systems.

- **Data Models**: Define how data is structured to represent real-world entities.
- **Databases**: Specify how data is stored on machines, enabling efficient retrieval and processing.
- **Key Insight**: Data models describe real-world data, while databases focus on machine-level storage and access.

## Data Sources

Data in ML systems comes from various sources, each with unique characteristics.

### User Input

- Includes text, images, videos, uploaded files, etc.
- Requires handling multimodal data (e.g., samples with both images and text).

### System-Generated Data

- Generated by system components, such as:
    - **Logs**: Record system state and events (e.g., memory usage, service calls, package usage).
    - **Model Outputs**: Predictions or results from ML models.

> [!note] Logs in ML Systems
> 
> - Logs provide visibility into system performance for debugging and optimization.
> - Common log contents:
>     - System state (e.g., memory usage, number of instances).
>     - Job results (e.g., batch processing, model training outcomes).
> - **Challenges**:
>     - **Volume**: Logs grow rapidly due to extensive logging in ML systems.
>     - **Signal-to-Noise Ratio**: Hard to find relevant information in large log volumes.
>     - **Storage**: Logs can be moved to low-access storage (e.g., Amazon S3) for cost savings if not frequently accessed.
> - **Solutions**:
>     - Log processing services (e.g., Logstash, Datadog, Logz.io) use ML to analyze logs.
>     - Retention policies: Store logs only as long as they are useful for debugging.

## Data Formats

Choosing the right data format is crucial for efficient storage and access, depending on future use cases.

### Key Questions for Data Storage

- How to store multimodal data (e.g., images and text together)?
- Where to store data for cost-effective, fast access?
- How to store complex ML models for compatibility across different hardware?

### Data Serialization

> [!definition] Data Serialization  
> The process of converting a data structure or object state into a format that can be stored or transmitted and reconstructed later.

### Common Data Formats

- **CSV (Comma-Separated Values)**:
    - **Row-major**: Consecutive elements in a row are stored next to each other in memory.
    - **Text-based**: Human-readable, stored as plain text.
- **Parquet**:
    - **Column-major**: Consecutive elements in a column are stored next to each other.
    - **Binary format**: Not human-readable, optimized for columnar data storage and retrieval.
- **JSON**:
    - **Text-based**: Human-readable, flexible for structured data.

> [!tip] Why Use Parquet?  
> AWS recommends Parquet because it is:
> 
> - Up to **2x faster** to unload compared to text formats.
> - Consumes up to **6x less storage** in Amazon S3.

## ETL: Extract, Transform, Load

ETL is a framework for preparing data for ML systems by processing it through three stages.

### Extract

- Retrieve data from various sources.
- Validate data and reject corrupted or malformatted entries.
- Notify data sources about rejected data.

> [!important] Importance of Extraction  
> Proper validation during extraction saves significant time in downstream processing.

### Transform

- Core data processing stage:
    - Join data from multiple sources.
    - Clean and standardize data (e.g., unify "Male"/"Female" to "M"/"F").
    - Perform operations like transposing, deduplicating, sorting, aggregating, or deriving new features.
    - Additional data validation.

### Load

- Store transformed data in the target destination (e.g., file, database, data warehouse).
- Decide frequency and method of loading (e.g., batch or real-time).

> [!example] ETL Example
> 
> - **Extract**: Pull user data from a CSV file and logs from a server.
> - **Transform**: Standardize gender values, remove duplicates, and calculate average user age.
> - **Load**: Store the processed data in a Parquet file in a data warehouse.

## Dataflow

Dataflow refers to how data moves between processes in a system.

### Modes of Dataflow

1. **Through Databases**:
    - Data is stored and queried in databases.
2. **Through Services**:
    - Data is passed via requests (e.g., REST or RPC APIs).
3. **Through Real-Time Transports**:
    - Data moves via systems like Apache Kafka or Amazon Kinesis.

### Data Passing Through Services

- **REST (Representational State Transfer)**:
    - Designed for network requests, common in public APIs.
- **RPC (Remote Procedure Call)**:
    - Mimics local function calls for remote services, often used within organizations.

> [!note] REST vs. RPC
> 
> - REST is suited for network-based APIs.
> - RPC is optimized for internal services within the same data center.

### Real-Time Transports (Event-Driven Architecture)

- Data broadcast to a real-time transport is called an **event**.
- **Event Bus**: A system (e.g., Kafka, Kinesis) that handles event distribution.
- **Pubsub (Publish-Subscribe)**:
    - Services publish events to topics; subscribers read all events in subscribed topics.
    - Producers and consumers are decoupled.
    - Retention policies (e.g., 7 days) manage data storage.
- **Message Queue**:
    - Similar to pubsub but often used for task queuing.

> [!example] Pubsub in Action  
> A service publishes user activity events to a Kafka topic. Multiple analytics services subscribe to the topic to process the events independently.

## Batch Processing vs. Stream Processing

Data processing can occur in two primary modes: batch and stream processing.

### Batch Processing

- Processes **historical data** stored in databases, data lakes, or data warehouses.
- Performed via **batch jobs** (e.g., daily computation of average ride surge charges).
- Leverages distributed systems like **MapReduce** or **Spark** for efficiency.

> [!definition] Batch Features (Static Features)  
> Features extracted via batch processing, typically for data that changes slowly (e.g., driver ratings).

### Stream Processing

- Processes **streaming data** in real-time transports (e.g., Kafka, Kinesis).
- Computations occur frequently (e.g., every 5 minutes) or on-demand (e.g., when a user requests a ride).
- Offers **low latency** by processing data as it arrives.
- Enables **stateful computation**, avoiding redundant processing of historical data.

> [!definition] Streaming Features (Dynamic Features)  
> Features extracted via stream processing, reflecting the current system state (e.g., number of available drivers).

> [!example] Stateful Stream Processing  
> To track user engagement over a 30-day trial:
> 
> - **Batch**: Recompute all 30 days daily.
> - **Stream**: Compute only new data daily and join with prior results.

### Combining Batch and Streaming Features

- Many ML systems require both batch and streaming features.
- **Example**: Predicting ride prices using:
    - **Batch Features**: Driver ratings (static, computed daily).
    - **Streaming Features**: Current driver availability, recent ride prices (dynamic, computed in real-time).

### Stream Processing Engines

- **Apache Flink**, **KSQL**, **Spark Streaming**:
    - Handle complex stream processing with joins, aggregations, and multiple data sources.
    - Flink and KSQL offer SQL abstractions for data scientists.
- **Kafka Streams**:
    - Limited for complex ML feature extraction due to simpler capabilities.

> [!important] Stream vs. Batch Processing
> 
> - Stream processing handles unbounded data with variable rates, making it more complex.
> - Batch processing is a special case of stream processing (e.g., Apache Flink can handle both).

## Key Takeaways

- **Data Models and Databases**: Structure real-world data and define machine-level storage.
- **Data Sources**: Handle user input and system-generated data (logs, model outputs).
- **Data Formats**: Choose CSV (row-major), Parquet (column-major), or JSON based on use case.
- **ETL**: Extract, transform, and load data to prepare it for ML.
- **Dataflow**: Use databases, services (REST/RPC), or real-time transports (Kafka/Kinesis).
- **Batch vs. Stream Processing**:
    - Batch for static features; stream for dynamic features.
    - Combine both for robust ML systems.
- **Tools**: Use Spark/MapReduce for batch, Flink/KSQL for streaming.
