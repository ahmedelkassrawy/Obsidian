DataPilot AI — Agent System Architecture

Text-to-SQL LangGraph agent · 15 nodes · ~8 conditional routing points · LangSmith evaluation
1 · Entry
POST /api/query validates + warms connection
Validates question and source_id. Warms connection string via data_source_service. Generates a thread_id.
→
AgentGraph.run() invoke StateGraph
Entry point. Constructs StateGraph(AgentState) with 18 fields. Exposes run() and resume(). Records query history in finally block.
2 · Router
Intent Router INQUIRE · ADD · UPDATE · DELETE
Regex fast-path first, then LLM fallback with INTENT_ROUTER_PROMPT (20 max_tokens). Defaults to INQUIRE on failure.
3 · Schema Intelligence
Fetch Schema SQLAlchemy Inspector
Ignores injected schema_service param. Filters Oracle system tables and MSSQL replication tables. Cached in _SCHEMA_CACHE.
→
Filter Schema LLM prune (≥10 tables)
LLM prunes to relevant tables for the query. Skips entirely if schema has ≤10 tables (context_filtering.py:13).
4 · Memory & Scenario
Load Memory LangGraph InMemoryStore
Queries up to 3 past query memories for this source_id from LangGraph's InMemoryStore.
→
Scenario Lookup FAISS similarity ≥ 0.45
Searches file-backed FAISS index over scenarios.md. Hash-based embedding (token → bucket mod 512, L2-normalized). Injects 10 most recent entries regardless of match.
5 · SQL Generation
READ INTENTS
Scenario Matched?
If FAISS match ≥ 0.45, pre-populates state.sql with the past solution, skipping generation entirely.
→
Generate SQL (SELECT) 28 rules · sanitize output
SQL_GENERATION_PROMPT with 28 rules covering DISTINCT, ORDER BY, aggregations, Arabic questions, fuzzy table matching. System: "raw SQL only, no markdown, no backticks". Sanitized via _sanitize_sql().
WRITE INTENTS
Generate Mod SQL ADD · UPDATE · DELETE
Uses intent-specific prompts: SQL_ADD_PROMPT, SQL_UPDATE_PROMPT, SQL_DELETE_PROMPT. Checks for mock output after generation.
6 · Approval (Write Intents)
Approval Node interrupt() via checkpointer
Only reached for write intents. CLI mode prompts stdin. API flow calls interrupt() — pauses graph execution. Route POST /query/approval resumes via Command(resume=approved).
✅ Approved → Execute SQL
·
❌ Denied → End
7 · Execution & Retry
Execute SQL validate · cache · run
Validates forbidden keywords (DDL always blocked, DML blocked for INQUIRE). Checks _STALE_CACHE for reads. DBService.run_query() fetches up to 1000 rows. On write success, invalidates schema cache.
↔
Fix SQL retry ≤ 3 attempts
On failure + retry_count < 3: constructs SQL_FIX_PROMPT with error message, asks LLM to rewrite SQL, re-executes. Alternates with sql_execution_node for attempts 2+.
⤴ retry loop: fix_sql_node ↔ sql_execution_node
✅ Success → Post Process
·
❌ Fail ≥ 3 → Scenario Failure → Lesson → Doc
8 · Post Process (Parallel)
Visualization auto-detect chart → Plotly JSON
Runs generate_visualization() which auto-detects chart type from query results and returns a full Plotly JSON spec.
Insights bilingual [{ar}, {en}]
Sends first 20 rows (5 columns) to LLM via INSIGHT_PROMPT. Returns array of {ar, en} objects. Falls back gracefully on empty results or parse failure.
Suggestions 3 follow-up questions
Sends question + SQL + schema + 3-row preview to LLM via SUGGESTION_PROMPT. Returns exactly 3 suggestions. Skipped for ≤1 result row.
⚡ ThreadPoolExecutor(max_workers=3) — runs concurrently
9 · Document & Persist
Document QueryDocument model
Constructs QueryDocument Pydantic model: question, sql, results, results_count, visualization, insights, suggestions, executed_at. Logged at INFO level.
→
Persist Memory InMemoryStore · query history
Saves query summary to LangGraph's InMemoryStore under namespace (source_id, "query_history").

⏳ Background Evaluation
Evaluation Service daemon thread · fire-and-forget
Spawned after graph.invoke() returns. Runs SQL syntax check (regex), LLM correctness judge, schema relevance. Score = correctness×0.4 + completeness×0.25 + efficiency×0.15 + schema×0.1 + syntax×0.1. Posts to LangSmith if LANGCHAIN_API_KEY set. Always non-blocking.
⚠ Architecture Notes
1. fetch_schema_context ignores injected schema_service param — dead async code exists but is never called by any graph node.
2. ANSWER_PROMPT defined in prompts.py but never invoked by any node. No natural-language answer phase.
3. Dual memory systems: ScenarioMemory (file + FAISS) and InMemoryStore both inject context into the SQL prompt — potential duplication.
4. SQL cache: plain module dict, no true LRU, no TTL. Writes invalidate schema cache but not the stale cache — cached reads survive data changes.
5. Arabic: single prompt rule with 15 keyword mappings. No preprocessing, transliteration, or separate model.
6. approval_store.py (Redis-backed TTL store) — dead code, never referenced by any node or route.
7. No token budget management. Full schema dumps can exceed context windows. Context filter skips ≤10 tables but the filter LLM call still gets the full text.
8. Evaluation is truly fire-and-forget — daemon thread, no signal in the response. Fails silently if LANGCHAIN_API_KEY is unset.
